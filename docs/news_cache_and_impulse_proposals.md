# Кэш новостей без БД, свежесть, макро/геополитика, расчёт новостного импульса

Набор **вариантов на выбор**; финальный план фиксируется отдельно после решения команды.

---

## 1. Кэш без базы: принципы

**Зачем кэш при скромном потоке yfinance:** не столько «накопить историю», сколько **не бить API** (Marketaux, NewsAPI, LLM) при повторных запросах в тестах и Telegram, и **дедуплицировать** один и тот же URL из разных вызовов.

**Что не кэшировать надолго:** «старые» новости для **торгового решения** действительно малоинтересны — TTL и объём кэша должны отражать **окно принятия решения** (например 6–48 ч), а не архив.

### 1.1. Ключи и уровни

| Уровень | Ключ (пример) | TTL (пример) | Назначение |
|--------|----------------|-------------|------------|
| **Сырой ответ провайдера** | `provider + hash(query_params)` или `url` для статьи | 5–30 мин | Повторный `get_news` / HTTP за минуты |
| **Нормализованный список** | `symbol + bucket_utc_hour` | 1–2 ч | Один «снимок» ленты на тикер за час |
| **Агрегат импульса** | `symbol + window_end + method_version` | 15–60 мин | Повторный расчёт без пересчёта LLM |

Хранилище: каталог **`./.cache/nyse/`** или `~/.cache/nyse/` — JSON/msgpack + метаданные `fetched_at`, `ttl_sec`.

### 1.2. Инвалидация «по свежести»

- При запросе: если `now - fetched_at > ttl` → перезагрузка.
- Жёсткий **cutoff по времени публикации** при любом использовании: `published_at >= now - NEWS_WINDOW_HOURS` (например 24–72 ч) — старое **не попадает** в импульс, даже если лежит в файловом кэше (можно чистить каталог раз в сутки cron/systemd timer).

### 1.3. Ограничение yfinance

- Лимит статей и задержка обновления — известны; кэш **не исправляет** скупость ленты, только экономит повторы.
- Для **максимально свежих** и **макро/геополитики** нужны **дополнительные каналы** (ниже), а не только кэш Yahoo.

## 2. Как получать более свежие и релевантные новости

Идея: **несколько источников** + **макро-календарь** как триггер «сейчас важно смотреть шире».

### 2.1. API (предпочтительно по стабильности)

| Источник | Свежесть / сильные стороны | Ограничения |
|----------|----------------------------|-------------|
| **Marketaux** | Глобальные заголовки, entity + sentiment | Квоты, платёж |
| **NewsAPI** | Тематические запросы, много СМИ | Лимиты free, не везде задержка минут |
| **Alpha Vantage** (при подписке) | Единый контракт | Лимиты на free |
| **RSS** центробанков / крупных агентств | Очень быстрые официальные тексты | Парсинг XML, не «поиск по тикеру» |
| **Investing лента** (как в lse) | Рыночные заголовки | Скрейп/HTML, хрупкость |

**Макро/геополитика:** отдельные **подписки RSS** (Reuters World, FT, официальные регуляторы) + опционально **ключевые слова** в NewsAPI/Marketaux (`Fed`, `OPEC`, `sanctions`, …) — без привязки к одному тикеру (см. `MACRO` / `US_MACRO` в lse).

### 2.2. Парсинг экономических сайтов

- **Плюс:** уникальный контент, иногда быстрее агрегаторов.
- **Минусы:** ToS, robots.txt, частые смены вёрстки, 403/429, юридические риски.
- **Практика в вашем стеке:** в **lse** уже есть паттерны (`investing_news_fetcher`, `investing_calendar_parser`) с backoff — **переиспользовать идею**, не размножать голый скрейп без fallback.

**Рекомендация:** приоритет **официальным RSS/API**; парсинг лент — **точечно** (1–2 сайта), с тестами и rate limit.

### 2.3. Связка с календарём

- За **N минут до/после** HIGH-события (CPI, FOMC, NFP) — усилить опрос **макро-каналов** и расширить окно `NEWS_WINDOW_HOURS` для **глобальных** новостей (не только по тикеру).
- Это не заменяет источники, но **направляет**, когда «максимально свежие» критичны.

## 3. Методы расчёта новостного импульса (на выбор)

Обозначения: статьи после нормализации с полями `sentiment` (если есть), `time`, опционально **`impact_channel`** (INCREMENTAL / REGIME / POLICY_RATES).

### 3.1. Простые и дешёвые (без LLM)

1. **Взвешенное среднее sentiment** по окну с весами \( w = e^{-\lambda \cdot age\_hours} \) — свежее важнее.
2. **Минимум / доля негатива:** доля статей с `sentiment < -0.3` — флаг «токсичный фон».
3. **Сумма по каналам:** отдельно агрегировать **REGIME** / **POLICY_RATES** (даже грубо по словарю ключевых слов до LLM) — не смешивать с тикерным потоком в одну среднюю.

### 3.2. Как у Kerima (LLM + агрегатор)

- Отбор статей (selection) → сигнал на статью (sentiment, impact, relevance, horizon, confidence) → **взвешенный `bias`** (`AggregatedNewsSignal`).
- **Плюс:** согласовано с pystockinvest; **минус:** токены и латентность.

### 3.3. Гибрид

- Числовой sentiment с **Marketaux** / эвристика по ключевым словам → **предфильтр** и первичный скалярный импульс.
- LLM **только** если |предимпульс| > порога или есть флаг **REGIME** / событие календаря.

### 3.4. Режимные события

- Если сработал классификатор **REGIME** (или ручной список источников/тегов) → ввести **`regime_overhang`** (0…1) и **снизить доверие** к чисто техническому сигналу или потребовать подтверждение второго канала (см. `news_calendar_inventory.md` §5.4–5.5).

## 4. Рекомендуемые комбинации (коротко)

| Профиль | Кэш | Источники свежести | Импульс |
|---------|-----|-------------------|---------|
| **MVP nyse** | Файлы, TTL 30–120 мин, окно 24–48 ч | Yahoo + календарь + при наличии ключа Marketaux/NewsAPI | Взвешенное среднее + доля негатива; REGIME по словарю |
| **Ближе к Kerima** | То же + кэш ответов LLM по хешу входа | + RSS макро | Selection + Signal LLM как в pystockinvest |
| **Экономия токенов** | Агрессивный кэш агрегатов | Гибрид: числа с API → LLM только при пороге | П. 3.3 |
